# Release 0.4.0 Closeout

**Status**: In Progress (2026-07-20)

## Objective

Publish `v0.4.0` as the next pre-1.0 minor release after `v0.3.9`, carrying the
user-visible hierarchical model picker, contextual status bar, and composer multiline
and scrollback-wrap work on `main` through the repository-standard tag-driven release
workflow.

## Version Decision

`v0.4.0` is a minor bump because the release adds user-visible TUI capabilities rather
than only correcting the already-published `v0.3.9` behavior. Talos remains pre-1.0;
this release does not change the REL-002 `NO-GO` decision or claim v1.0 readiness.

## Scope

- Synchronize the workspace package version and every internal path dependency to `0.4.0`.
- Update current README release references to `v0.4.0`.
- Run `./scripts/release_preflight.sh v0.4.0` with the pinned Rust 1.97.0 toolchain.
- Create and push one immutable annotated `v0.4.0` tag after preflight passes.
- Let `.github/workflows/release.yml` own the asynchronous platform builds, checksums,
  and GitHub Release publication after the tag is pushed.
- Confirm workflow handoff without waiting for CI completion, per maintainer direction.

## Non-Goals

- No crates.io publication.
- No tag rewrite, force-push, or reuse after failure.
- No `v1.0.0` readiness claim; REL-002 remains NO-GO.
- No claim that I142/TUI-032 passed real-terminal acceptance while its owner documents
  remain in Review.
- No manual release asset upload while the tag-driven workflow is operating normally.

## Release Content Boundary

GitHub Release notes are generated from commits after `v0.3.9`. The release includes
MODEL-007 hierarchical model/variant selection, TUI-031 contextual workspace status,
and I142/TUI-032 composer multiline and explicit finalized-history wrapping. The
keyboard protocol remediation is covered by automated tests and portable `Ctrl+J`
fallback; direct-terminal acceptance remains tracked by I142.

## Validation Plan

- `cargo metadata --locked --no-deps --format-version 1` reports one Talos version:

- `./scripts/release_preflight.sh v0.4.0`.

## Failure And Recovery

- Preflight failure: do not create the tag; fix on `main` and rerun.
- Workflow failure after tag push: keep `v0.4.0` immutable, record the failure, fix the
  source, and publish a new patch version such as `v0.4.1`.
- External GitHub outage: record the pushed commit/tag and workflow blockage without
  claiming a completed GitHub Release.
