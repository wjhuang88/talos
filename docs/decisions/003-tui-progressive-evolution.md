# ADR-003: TUI 渐进式演进策略

## 状态

已接受 (Accepted)

## 背景

Talos 需要一个终端用户界面 (TUI) 来提供交互式体验。初期讨论中，TUI 被规划在 I009（最后一个迭代）作为"打磨"阶段一次性实现。

这种"最后集中实现"的方式存在以下问题：

1. **反馈延迟**：直到 I009 才有可视化界面，前 8 个迭代的交互体验无法验证
2. **集成风险**：一次性引入大量 TUI 代码，与已有功能集成时可能产生冲突
3. **测试盲区**：压缩、会话分支、技能加载等功能在纯 CLI 模式下难以直观验证
4. **用户体验断层**：从 I001 到 I008 都是命令行工具，I009 突然变成全功能 TUI，用户需要重新学习

## 决策

采用**渐进式演进策略**：从 I005 开始引入基础 TUI 壳，每个后续迭代增量添加对新功能的可视化支持。

### 演进路线

| 迭代 | TUI 新增能力 | 验证场景 |
|------|-------------|---------|
| **I005** | 基础壳：聊天视口 + 输入区 + 状态栏 + Ctrl+C + 流式输出 | 用 Mock LLM 测试压缩时 TUI 不卡顿 |
| **I006** | 工具调用气泡 + 审批覆盖层 (y/a/n) + 会话列表 | 权限提示直接在 TUI 中弹出 |
| **I007** | 技能索引侧栏 + `/model` 切换 + 系统提示词预览 | 加载 SKILL.md 后 TUI 显示技能列表 |
| **I008** | 进化洞察面板 + `/learned` 命令 + 置信度可视化 | 自进化学习后 TUI 显示学到的模式 |
| **I009** | MCP 工具标记 + 插件状态指示器 + Hook 日志 | MCP 工具在 TUI 中有特殊标识 |
| **I010** | Nord 主题 + markdown 渲染 + diff 显示 + steering 队列 + slash 命令 + Guardian + headless + policy DSL | 发布级打磨 |

### 技术选型

- **框架**：ratatui + crossterm（已在 REFERENCE-PROJECTS.md §18 确认）
- **主题**：Nord 配色方案（已在 REFERENCE-PROJECTS.md §19 确认）
- **架构**：独立 `talos-tui` crate，从 I005 开始引入

### 设计原则

1. **最小可用壳**：I005 的 TUI 只包含聊天视口、输入区、状态栏，不追求美观
2. **功能驱动增量**：每个迭代的 TUI 改进都由该迭代的新功能驱动（如 I006 的审批需要覆盖层）
3. **向后兼容**：保留 `-p`（print mode）和 `--no-tui` 选项，允许用户选择纯 CLI 模式
4. **Mock LLM 测试**：I005-S1 实现的 Mock LLM 用于测试 TUI 在各种场景下的表现（长对话、工具调用、错误处理）

## 后果

### 正面

- **早期反馈**：从 I005 开始就能直观验证交互体验
- **降低风险**：TUI 代码分散在 6 个迭代中，每次增量小，易于调试
- **测试覆盖**：每个新功能都有对应的 TUI 可视化验证
- **平滑过渡**：用户从 I005 开始逐步适应 TUI，而非突然切换

### 负面

- **复杂度增加**：需要维护 TUI 状态与后端状态的同步
- **迭代工作量增加**：每个迭代多一个 TUI story（约 M 级别）
- **早期 TUI 简陋**：I005-I009 的 TUI 不够美观，可能影响演示效果

### 缓解措施

- **Mock LLM**：I005-S1 实现的 Mock LLM 可以模拟各种场景，减少真实 API 调用
- **TUI 测试框架**：从 I005 开始建立 TUI 测试基础设施，确保增量不破坏已有功能
- **Nord 主题预留**：I005 的 TUI 壳使用 Nord 配色常量，I010 只需调整细节
